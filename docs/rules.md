# MATLAB 函数定义与注释规范（项目规则）

本规范用于统一项目内 MATLAB 函数的“定义、帮助注释、示例、变更记录”写法。  
目标：**专业、可读、可维护、可自动生成帮助信息**，整体风格对齐 MathWorks 官方函数文档格式（`help` 可直接阅读）。

---

## 1. 适用范围

- 适用于项目内所有 `.m` 函数文件（公共函数、工具函数、脚本入口函数）。
- 新增函数必须遵守本规范；历史函数在修改时逐步补齐。

---

## 2. 文件与函数基础规则

- 一个主函数对应一个同名文件：`functionName.m`。
- 主函数定义行后紧跟 `H1` 行（全大写函数名 + 一句话摘要）。
- 公共函数必须提供完整 `help` 注释块。
- 禁止无说明的“魔法值”；关键默认值需写入注释和参数解析器。
- 函数内应进行输入校验（`validateattributes` / 自定义校验）。

---

## 3. 标准注释结构（强制）

每个公共函数按以下顺序组织注释：

1. **H1 行**：`%FUNCNAME 一句话功能摘要`
2. **语法（Syntax）**
3. **功能描述（Description）**
4. **输入参数（Input Arguments）**
5. **可选参数（Name-Value）**
6. **输出参数（Output Arguments）**
7. **异常与边界行为（Errors/Warnings）**
8. **使用示例（Examples）**
9. **参见（See also）**
10. **元信息（作者、版本、日期、变更记录）**

> 说明：顺序尽量固定，便于统一阅读和自动检查。

---

## 4. 推荐模板（可直接复制）

```matlab
function [out1, out2] = functionName(in1, varargin)
%FUNCTIONNAME 一句话功能摘要（动词开头，明确对象）
%
% 语法:
%   out1 = functionName(in1)
%   [out1, out2] = functionName(in1, 'Name', Value, ...)
%
% 功能描述:
%   用 2-5 行说明函数做什么、适用场景、核心处理逻辑。
%
% 输入参数:
%   in1            - 参数说明（类型、单位、约束）
%
% 可选参数（名值对）:
%   'OptionA'      - 说明，默认: xxx
%   'OptionB'      - 说明，默认: xxx，可选值: {'A','B'}
%
% 输出参数:
%   out1           - 输出说明（类型、维度）
%   out2           - 输出说明
%
% 异常与边界行为:
%   - 参数非法时抛出 error: <标识/条件>
%   - 数据缺失时给出 warning: <行为>
%
% 示例:
%   r = functionName(data)
%   [r1, r2] = functionName(data, 'OptionA', true)
%
% 参见: RELATEDFUNC1, RELATEDFUNC2
%
% 作者: blue.ge(葛维冬@Smart)
% 版本: 1.0
% 日期: 2026-04-09
% 变更记录:
%   2026-04-09 v1.0  首次创建。

% ===== 参数解析与校验 =====
p = inputParser;
addParameter(p, 'OptionA', true, @islogical);
parse(p, varargin{:});

validateattributes(in1, {'double','single'}, {'nonempty'}, mfilename, 'in1');

% ===== 主逻辑 =====
out1 = in1;
out2 = [];
end
```

---

## 5. 各部分写作要求

### 5.1 H1 行

- 必须首行出现，格式：`%FUNCNAME 摘要`
- 摘要不超过 25 字，突出“动作 + 对象”。

### 5.2 输入/输出参数说明

- 每个参数写清：**类型、维度、取值范围、单位（如有）**。
- Name-Value 参数必须写“默认值 + 可选值 + 含义”。

### 5.3 示例（至少 2 个）

- 最少提供：
  - 一个最简调用；
  - 一个带 Name-Value 的典型调用。
- 示例应可在当前项目上下文中直接理解和复现。

### 5.4 错误与警告

- 预期失败路径要明确：什么情况下 `error`，什么情况下 `warning`。
- 报错信息应包含函数名与关键上下文，便于定位问题。

### 5.5 参见（See also）

- 填写项目内关联函数或 MATLAB 官方相关 API（全大写函数名）。

### 5.6 变更记录

- 每次重要修改需更新：`日期 + 版本 + 变更摘要`。
- 推荐语义化版本：`1.0`（首版）、`1.1`（兼容增强）、`2.0`（不兼容变更）。

---

## 6. 代码实现配套规范

- 参数解析统一用 `inputParser`（或项目统一解析器）。
- 参数校验统一使用 `validateattributes`/`mustBe*`。
- 复杂逻辑拆分为本地子函数，子函数命名建议 `i_xxx`。
- 对外函数避免副作用；若会写文件/修改模型，需在注释中显式声明。
- 默认行为必须“安全可重复”（例如 `overwrite=false` 默认不覆盖）。

## 8. 与现有函数对齐建议

以 `findCalibParams.m` 为风格基线，建议统一改进：

- 保持现有“输入参数/可选参数/输出参数/示例”结构；
- 增加“异常与边界行为”“变更记录”两段；
- 名值参数默认值与约束写全；
- 关键行为（如是否跳过 mask/lib、搜索深度）在注释中写明确。

## 9. 建议的函数头最小必填项（强制）

以下字段缺一不可：

- `H1 行`
- `语法`
- `输入参数`
- `可选参数（如有）`
- `输出参数`
- `示例`
- `作者/版本/日期/变更记录`

## 10. 开源化目录组织与命名规范（新增）

为将当前工程逐步演进为“专业开源项目”，在不破坏现有功能前提下，建议按以下规则进行目录与命名治理。

### 10.1 顶层目录建议（目标结构）

建议逐步收敛为：

- `src/`：核心 MATLAB 源码（函数、类、业务逻辑）
- `examples/`：可运行示例与 demo
- `tests/`：自动化测试（`matlab.unittest`）
- `docs/`：设计文档、规范、架构说明
- `data/`：样例输入数据（小体积、可公开）
- `tools/`：构建、发布、检查脚本
- `resources/`：图片、模板、静态资源
- `third_party/`：外部依赖（含许可证说明）

> 迁移建议：保留现有目录，新增目标目录并逐步迁移；避免一次性大规模重构。

### 10.2 当前工程的优先优化点（结合现状）

- 目录名中存在空格（如 `blue script`），建议新目录统一使用小写下划线：`blue_script` 或并入 `src/`。
- 生成产物混入源码目录（如 `arxml_models_*`、`*_PortsInfo_*.xlsx`），建议迁移到：
  - `artifacts/`（本地生成产物）
  - `cache/`（模型缓存）
  - 并加入 `.gitignore`
- 目录大小写不统一（如 `lib` 与 `Lib` 共存），建议统一小写目录名，避免跨平台问题。
- 文档分散在多处 README，建议补充顶层 `docs/` 索引页并统一链接导航。

### 10.3 MATLAB 源码组织规则

- 函数代码放置于 `src/+pkg/...` 或 `src/module/...`，避免长期平铺。
- 类定义统一放置于 `src/class/@ClassName/`（或按包命名空间组织）。
- “查找/创建/修改/删除”类工具函数建议按能力拆分子目录并统一前缀：
  - `find*`, `create*`, `change*`, `delete*`
- 脚本入口与函数实现分离：
  - 入口脚本放 `examples/` 或 `tools/`
  - 可复用逻辑必须封装为函数（放 `src/`）

### 10.4 文件命名规范

- 统一使用 ASCII 与小写风格，建议：
  - 目录：`lower_snake_case`
  - 文件：MATLAB 函数名与文件名严格一致（`findPortsInfoToGjson.m`）
- 禁止在同一项目混用仅大小写区分的名称。
- 自动生成文件建议包含稳定前缀与时间戳，例如：
  - `ccm_internal_swc_ports_info_YYYYMMDD_HHMMSS.xlsx`
- 临时文件统一前缀：`tmp_`，并放入 `artifacts/tmp/`。

### 10.5 文档与开源元信息（建议必备）

开源最小集合建议补齐：

- 顶层 `README.md`（安装、快速开始、目录说明、示例）
- `LICENSE`（许可证）
- `CONTRIBUTING.md`（贡献流程）
- `CODE_OF_CONDUCT.md`（行为准则）
- `CHANGELOG.md`（版本变更记录）

### 10.6 生成产物与版本控制规则

- 以下类型默认不入库（除非有明确评审结论）：
  - `*.slx` 生成缓存（批量导入产生）
  - `*_PortsInfo_*.xlsx`、中间导出表
  - 临时日志、缓存目录
- 对需要共享的“基线样例”可放 `data/samples/`，并提供来源说明。

### 10.7 渐进式迁移策略（推荐）

1. 先新增目标目录（`src/tests/examples/tools/artifacts`）。
2. 新增代码按新规范落位，旧代码保持兼容路径。
3. 为关键旧路径添加过渡包装函数（deprecate 注释）。
4. 分批迁移模块并补测试，最后清理旧目录。

### 10.8 目录组织检查清单

- [ ] 顶层目录职责清晰、无明显混放  
- [ ] 源码目录与产物目录分离  
- [ ] 目录大小写与命名风格统一  
- [ ] 自动生成文件不直接进入源码目录  
- [ ] 关键开源元信息文件齐全  
- [ ] 新增模块默认进入 `src/`，示例进入 `examples/`
